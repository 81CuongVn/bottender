Session

介紹

對話可能會發生在 1 對 1 的私訊或甚至是一個有很多人在裡面的頻道中。 正因為如此，Bottender 了一個「session」機制來分辨不同對話並儲存暫時性的資料在對應的對話上。 這就類似 session 在典型的網頁伺服器中的運作方式。 以下是在你使用 session 時可能會想知道的幾個詞彙：

  • Session state：在對話上的暫存資料。
  • Session id：對話的唯一識別符。
  • Session storage：儲存 session 的地方。

Session State

State 被廣泛地使用在日常裝置的流程控制，例如：紅綠燈。 State 的改變是反應外部的輸入且/或滿足了某些條件。 例如：綠燈 （state）會在六十秒後（滿足的時間條件）轉變成黃燈。

計數機器人範例

試想一個計數用的機器人它回應它所收到的訊息事件總數，例如：

You > Hi
Bot > Count: 1
You > Hi
Bot > Count: 2

在這個章節，我們會解釋如何正確地使用 state 來建立一個記數器。 它從一個計數用的 state 變數開始，這個變數會在機器人收到事件時被更新。

讓我們從一個像這樣的靜態 action 開始：

async function EventCount(context) {
  await context.sendText('Count: 1');
}

把 State 添加到對話上

  1. 在 bottender.config.js 檔案裡設定初始 state：
module.exports = {
  initialState: {
    count: 0,
  },
};

在添加了這個之後，Bottender 會針對新的對話 session 把初始 state 設定成一個特定的物件。

  1. 使用 context.state 取用 state 的值：
async function EventCount(context) {
  const count = context.state.count + 1;
  await context.sendText(`Count: ${count}`);
}

即使我們已經使用了 state 變數來呈現訊息內容，它依然拿到 Count: 1 作為結果。 這是為什麼我們會需要步驟 3。

  1. 使用 context.setState() 來設定 state 的值：
async function EventCount(context) {
  const count = context.state.count + 1;

  context.setState({
    count,
  });

  await context.sendText(`Count: ${count}`);
}

然後，他就如所預期的一般運作了。

注意：不要使用 this.state.stateKey = stateValue; 來調整 state 的值，這或許會在未來造成一些不預期的行為。

State 更新會被合併

當你呼叫 setState()，Bottender 把你提供的物件跟當前的 state 做淺層的合併。

context.state; // { count: 1 }

context.setState({
  myObject: {},
});

context.state; // { count: 1, myObject: {} }

但如果你想要做更深的合併，你必須明確地合併它：

context.setState({
  myObject: {
    ...context.myObject,
    key: value,
  },
});

在 Console 模式除錯 State

在 「Console 模式」 裡，你可以利用便利的內建指令 - /state 來幫助你除錯你的 state 變化：

You > /state

這個指令會用 JSON.stringify() 格式化 state 並把這個結果當作機器人訊息發送給你：

Bot > { "count": 1 }

Session ID

每個 session 都有一個獨一無二的識別符在上面。 你可以透過 session.id 取用它。 為了方便展示，下面的程式碼把平台以及 session id 資訊傳送給使用者：

module.exports = async function App(context) {
  await context.sendText(`context.platform: ${context.platform}`);
  await context.sendText(`session.id: ${context.session.id}`);
};

為了避免平台之間的 ID 衝突，Bottender 添加了平台名稱的前綴到 session 的 key 上面。

舉例來說，在 Telegram 上是 telegram:16423... 這樣、在 LINE 上是 line:U4af4980629... 這樣。

在個平台之間，session 的 key 本身可以非常不同。 它可能來自使用者 id、頻道 id、群組 id、房間 id 或任何可以用來區別對話的東西。

注意：在 LINE platform 上有三種類型的來源，被定義在 LINE 官方文件：user、group 以及 room。

如果請求是來自一個使用者，那 session 的 key 會是 source.userId，例如：line:U4af4980629...。 如果請求是來自一個群組，那 session 的 key 會是 source.groupId，例如：line:Ca56f94637c...。 如果請求是來自一個房間，那 session 的 key 會是 source.roomId，例如：line:Ra8dbf4673c...。

Session Storage

因為基於 HTTP 的應用程式是無狀態的，session 的任務是在對話期間儲存使用者的對話情境。 Bottender 提供數種立即可用的 session 驅動，例如：memory、file、redis 以及 mongo。 這些驅動可以透過富表達性且一致的 API 去取用。

Session 的資訊可以藉由 connector 來存取，例如 Session Id。此外，不管這個 session 是介於機器人跟一個使用者之間還是一群組用者之間都可以。 一個機器人可以同時支援多個通訊平台，而一個 connector 對應到一個通訊平台。

你可以指定一個明確的 session 過期時間來重置整個 state。 這讓機器人可以在對話後一段時間忘記（初始化）state，讓它行為更像人類。

設定 Session 驅動

在 bottender.config.js 檔案裡，可以透過 session.driver 的值來設定 Session 驅動：

// bottender.config.js

module.exports = {
  session: {
    driver: 'memory',
    stores: {
      memory: {
        maxSize: 500,
      },
      file: {
        dirname: '.session',
      },
      redis: {
        port: 6379,
        host: '127.0.0.1',
        password: 'auth',
        db: 0,
      },
      mongo: {
        url: 'mongodb://localhost:27017',
        collectionName: 'sessions',
      },
    },
  },
};

以下是四個可以使用的驅動：

  • memory - session 透過 LRU cache 被儲存在記憶體裡，沒有持久化。
  • file - sessions are stored in files placed in .session directory by default.
  • redis - session 被儲存在 redis 資料庫。
  • mongo - session 被儲存在 mongo 資料庫。

根據 NODE_ENV 使用不同的 Session 驅動

為了更短的迭代，建議開發時使用 memory 作為 session 驅動。 透過重新啟動程式，整個 session store 可以被完全重置。

你可以透過環境變數來決定哪一個 session 驅動要用在開發、哪一個 session 驅動要用在生產環境：

// bottender.config.js

module.exports = {
  session: {
    driver: process.env.NODE_ENV === 'production' ? 'mongo' : 'memory',
    // ...
  },
};

設定 Session 過期時間

預設情況下，Bottender 儲存的 session 不會過期。 要設定明確的過期時間的話，可以添加 session.expiresIn 設定到你的 bottender.config.js 檔案：

// bottender.config.js

module.exports = {
  session: {
    driver: 'memory',
    expiresIn: 15, // 15 minutes
    // ...
  },
};

透過上面的設定，使用者會在閒置了 15 分鐘以後，拿到一個新的 session 以及新的 session state。
