Sending Messenger Messages

發送文字訊息

文字訊息 是在所有通訊平台中最常見以及最常使用的訊息類型 。 當你想攜帶一些動態資料時，它提供了最小型的格式，例如：股價和天氣資訊。

async function SendHi(context) {
  await context.sendText('Hi!');
}

發送 Rich Media 訊息

Rich media 包含了圖片、影片、音訊以及檔案。

當你的目標放在抓住使用者的注意力時，例如：限量的促銷，rich media 非常有用。 此外，在要建立一個沈浸式的體驗時，例如：講故事，也很方便。

在下面的章節，你會看到四種不同的發送 rich media 訊息的方法。

如果你想要利用 Facebook 的快取，例如，更快的載入時間，那麼你就應該跳到章節 Attaching Saved Assets。

方法1：透過 URL 發送

透過 URL 發送可能是最簡單的實作方法。 我們推薦你在打造原型時使用這個方法；這個載入速度可能會讓你的用戶抓狂。

await context.sendImage('https://example.com/image.jpg');
await context.sendVideo('https://example.com/video.mp4');
await context.sendAudio('https://example.com/audio.mp3');
await context.sendFile('https://example.com/receipt.pdf');

方法2：Attaching Saved Assets

如果你想利用 Facebook 的快取，例如，令人愉悅的載入速度，你應該嘗試這個稍微複雜一點的方法。

首先，你需要透過 Attachment Upload API 取得一個 page-scoped Attachment Id。

一但你取得了 Attachment Id，你就可以透過以下的程式碼發送 rich media 訊息。

await context.sendImage({ attachmentId: '<ATTACHMENT_ID>' });
await context.sendVideo({ attachmentId: '<ATTACHMENT_ID>' });
await context.sendAudio({ attachmentId: '<ATTACHMENT_ID>' });
await context.sendFile({ attachmentId: '<ATTACHMENT_ID>' });

注意：Attachment Id 是隸屬於粉絲專頁的。

一個商業機器人專案通常會涉及一個開發環境用的 staging 粉絲專頁和一個正式環境用的 production 粉絲專頁。 在這個案例中，在發布機器人之前，你必須重新上傳所有的附件而且更新所有的 Attachment Id。

方法 3: Attaching ReadStreams

在接下來的範例中，你可以透過建立 readable stream 來發送 rich media。 The file path is relative to your bot location.

const fs = require('fs');

await context.sendImage(fs.createReadStream('image.jpg'));
await context.sendVideo(fs.createReadStream('video.mp4'));
await context.sendAudio(fs.createReadStream('audio.mp3'));
await context.sendFile(fs.createReadStream('receipt.pdf'));

方法 4: Attaching Buffers

在接下來的範例中，你可以透過建立 buffer 來發送 rich media。 這裡的檔案路徑是相對於你的機器人位置。

await context.sendImage(imageBuffer, { filename: 'image.jpg' });
await context.sendVideo(videoBuffer, { filename: 'video.mp4' });
await context.sendAudio(audioBuffer, { filename: 'audio.mp3' });
await context.sendFile(fileBuffer, { filename: 'receipt.pdf' });

發送 Template 訊息

簡單來說，template 訊息 是一個由圖片、影片、標題、子標題和按鈕所組成的互動式畫廊。

Template 訊息 是提供 rich media 互動的關鍵。 這常被使用在顯示多個選項以及下一步的動作給使用者的情境，例如：套用優惠券、預定房間、預約。

注意：如果你熟悉 LINE 和 Messenger，你會找到不同的聊天介面方法。 LINE 建立類似 HTML 超級彈性的聊天介面，Flex Messages，Messenger 則是傾向於專注在普通聊天介面可提供一致的用戶體驗。

Generic Template

In a Generic Template, you can create up to 10 items in a row. 每個項目包含了標題、子標題、圖片和最多 3 個按紐。

Note: Please refer to Messenger's official guide of Generic Template to find out the latest specification.

await context.sendGenericTemplate([
  {
    title: "Welcome to Peter's Hats",
    imageUrl: 'https://petersfancybrownhats.com/company_image.png',
    subtitle: "We've got the right hat for everyone.",
    defaultAction: {
      type: 'web_url',
      url: 'https://peterssendreceiveapp.ngrok.io/view?item=103',
      messengerExtensions: true,
      webviewHeightRatio: 'tall',
      fallbackUrl: 'https://peterssendreceiveapp.ngrok.io/',
    },
    buttons: [
      {
        type: 'postback',
        title: 'Start Chatting',
        payload: 'DEVELOPER_DEFINED_PAYLOAD',
      },
    ],
  },
]);

Button Template

Button Template 有點像 Generic Template；他們唯一的差別是移除了項目圖片。

注意： 請參考 Messenger 官方指南 Button Template 當中的最新規格。

await context.sendButtonTemplate('What do you want to do next?', [
  {
    type: 'web_url',
    url: 'https://petersapparel.parseapp.com',
    title: 'Show Website',
  },
  {
    type: 'postback',
    title: 'Start Chatting',
    payload: 'USER_DEFINED_PAYLOAD',
  },
]);

Media Template

你會發現 Media Template 的簡約和優雅。 如果你同意「一圖勝千言」就嘗試一下。它和 Generic Template 的差別在於移除了標題和子標題。

注意： 請參考 Messenger 官方指南 Media Template 當中的最新規格。

await context.sendMediaTemplate([
  {
    mediaType: 'image',
    attachmentId: '1854626884821032',
    buttons: [
      {
        type: 'web_url',
        url: 'https://en.wikipedia.org/wiki/Rickrolling',
        title: 'View Website',
      },
    ],
  },
]);

發送快速回覆

我們建議你將快速回覆視為使用者輸入文字的一種替代方式。

快速回覆是由最多 13 個 快速回覆按紐 所組成。 當使用點擊一個快速回覆按鈕，該快速回覆會消失。 接著，被點擊的按鈕的標題會被發在對話中作為回應。 在此同時，一則包含按鈕標題和額外 payload 的訊息會發送到你的 webhook。

在接下來的章節，你可以看到三種不同類型的 快速回覆。

注意：

  • 當我們第一次遇到快速回覆時，我們對於使用快速回覆或按紐的最佳時機感到困惑。 最後，我們發現快速回覆是引導用戶保持與機器人互動的最佳解決方案，而 URL 按紐 則帶領用戶到機器人之外。 更重要的是，快速回覆會在點擊快速回覆按鈕後消失，這使得快速回覆只在當前 context 有效，而按鈕則隨時可以在聊天紀錄中被觸發。
  • 注意： 請參考 Messenger 官方指南 快速回覆 當中的最新規格。

文字快速回覆

你可以在快速回覆中發送最多 13 個文字快速回覆按鈕。 每個文字快速回覆可以在按鈕標題旁加上一個額外的圖示。

await context.sendText('Hi!', {
  quickReplies: [
    {
      contentType: 'text',
      title: '<BUTTON_TEXT>',
      payload: '<DEVELOPER_DEFINED_PAYLOAD>',
    },
  ],
});

用戶電話號碼快速回覆

用戶電話號碼快速回覆可被視為同意蒐集用戶電話號碼按鈕。

Messenger 會自動帶入用戶電話號碼到快速回覆上。 許多 CRM(顧客關係管理) 使用電話號碼作為唯一 ID，未來做用戶綁定時會很方便。

注意：如果用戶個資沒有填過電話號碼，這個快速回覆就不會顯示。 而且，機器人只會在用戶點擊了快速回覆後才會收到電話號碼。

await context.sendText('Hi!', {
  quickReplies: [
    {
      contentType: 'user_phone_number',
    },
  ],
});

用戶電子信箱快速回覆

用戶電子信箱快速回覆可被視為同意蒐集用戶電子信箱按鈕。

Messenger 會自動帶入用戶電子信箱到快速回覆上。 許多 CRM(顧客關係管理) 使用電子信箱作為唯一 ID，未來做用戶綁定時會很方便。

注意：如果用戶個資沒有填過電子信箱，這個快速回覆就不會顯示。 而且，機器人只會在用戶點擊了快速回覆後才會收到電子信箱。

await context.sendText('Hi!', {
  quickReplies: [
    {
      contentType: 'user_email',
    },
  ],
});

Sending with Persona

為了讓用戶清楚了解到現在是真人客服還是機器人在代表這個粉絲專頁，Persona API 允許在對話中創造虛擬角色。 請參考關於 Persona 的文件。

await context.sendText('Hi!', {
  personaId: '<PERSONA_ID>',
});

頻率限制

如果你想使機器人的流量突然增加，例如，黑色星期五活動機器人，在你收到 error code 613 之前，你應該處理頻率限制問題。

粉絲專頁頻率限制是為了防止惡意行為和不良的用戶體驗。 對於有很多粉絲的大型粉絲專頁，Messenger 建議每分鐘發送 250 個請求。

頻率限制 會隨著粉絲專頁受眾的大小而改變。

24小時之內的發送量 = 200 * 受眾人數

為了防止觸及頻率限制，Messenger 也建議我們設計系統時考慮分散任何突然的大量負載。

注意：參考 Messenger 的官方文件頻率限制當中的最新規格。
